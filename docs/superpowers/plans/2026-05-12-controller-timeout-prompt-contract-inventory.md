# Controller Timeout Prompt Contract Registry

See also: [Docs index](../../index.md).

This registry tracks prompt contracts that are easy for a model to misread unless the model-facing wording defines the operational behavior. It is the source of truth for cleanup work across Reference ReAct prompts and Terminal-Bench preambles.

| Contract | Canonical wording | Model-facing surfaces | Failure mode | Required regression coverage | Status |
| --- | --- | --- | --- | --- | --- |
| bounded controller | Controller source must enforce an internal deadline shorter than the tool timeout, max command count, max move count, max work item count, max no-progress iterations, parser fail-closed behavior, and completion packet on every exit path. | Dev-shell include; deliberator prompt; Terminal-Bench CLI and protocol preambles; current-turn blocking facts. | Long controller consumes the tool timeout, leaves the required artifact missing, then gets rerun unchanged or after only syntax checks. | `reference-react-prompt-rules.test.ts` prompt parity; `test_cli_task_runner.py` and `test_kestrel_protocol.py` preamble parity; `context-builder-budget.test.ts` timeout and no-packet fact. | Implemented. |

## Audit Rule

When a prompt uses a quality adjective such as bounded, compact, semantic, observable, complete, durable, blocked, or safe, add a nearby operational definition and a regression test that proves the model-facing context contains that definition at the moment it matters.

Also avoid slash-shaped prose such as `command/move/work-item` in task-facing text. It makes task instructions harder for the model to read and can look like a path-shaped requirement. Prefer plain noun lists such as "command count, move count, or work item count."
