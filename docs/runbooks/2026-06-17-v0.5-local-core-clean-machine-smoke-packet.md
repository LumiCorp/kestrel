## Clean Profile Prerequisites

- Use a real clean macOS user profile or clean macOS machine.
- Do not use the repo checkout, `pnpm`, repo `.env`, Docker, Homebrew Postgres, or host Postgres for any smoke step.
- Ensure `~/Library/Application Support/Kestrel` and `~/Library/Application Support/@kestrel/desktop` are absent before the first fresh-profile launch, or record any existing state and use a new profile.
- Have an OpenRouter API key available.
- Install and start Ollama for the local-provider beta path.
- Prepare one small local workspace folder safe for test sessions.
- Extract the CLI tarball into a neutral folder, for example `~/KestrelSmoke/cli`.
- When testing hostile inherited env, set `DATABASE_URL` only for that shell and record it as intentionally hostile.

## Smoke Checklist

1. Fresh macOS profile launch
   - Unzip the Desktop artifact.
   - Launch `Kestrel.app` from the clean profile.
   - Pass: Desktop opens from the packaged artifact and creates or attaches to the default Local Core home at `~/Library/Application Support/Kestrel`.
   - Fail: Desktop requires a repo checkout, repo `.env`, `pnpm`, host Postgres, Docker, or Homebrew Postgres.

2. Desktop-first Local Core startup
   - Capture Desktop diagnostics/status.
   - Confirm one Local Core home, one Core API socket, one lock owner, and one database mode.
   - Pass: Desktop reports Local Core readiness and the lock owner is the Core daemon, not a shell-private Desktop store.

3. OpenRouter hosted-provider setup
   - Select OpenRouter, enter the API key, choose the test workspace, and start a first session.
   - Pass: setup persists and the session reaches a useful assistant response or an actionable provider error.

4. Ollama local-provider beta path
   - Start Ollama, select the Ollama provider, choose the test workspace, and start a first session.
   - Pass: Desktop does not request an API key and either uses the local model or reports a clear local-provider beta readiness issue.

5. Add workspace and start first session
   - Add or select the prepared workspace from Desktop.
   - Start a session and record workspace/session identifiers shown by Desktop.
   - Pass: workspace and session state are visible through Desktop diagnostics and remain Core-backed.

6. Quit/restart persistence
   - Quit Desktop, relaunch it, and re-open settings/workspace/session history.
   - Pass: provider selection, workspace, and session history persist through Local Core without shell-private durable state.

7. Extract CLI tarball and run status
   - Run:
     ```sh
     mkdir -p ~/KestrelSmoke/cli
     tar -xzf apps/cli/out/kestrel-cli-0.5.0-beta.0-darwin-arm64.tar.gz -C ~/KestrelSmoke/cli
     ~/KestrelSmoke/cli/bin/kestrel status
     ```
   - Pass: `kestrel status` reports Kestrel Local Core and the same Core home as Desktop.

8. CLI sees Desktop-created state without repo dependencies
   - From a shell without `KESTREL_HOME`, without manual `DATABASE_URL`, and outside the repo checkout, run packaged CLI status and any available workspace/session inspection command.
   - Pass: CLI sees Desktop-created Core state and does not require source checkout shims or repo `.env`.

9. CLI-first then Desktop attach path
   - Reset to a fresh profile.
   - Extract CLI, run `bin/kestrel status`, and perform the available CLI workspace/session setup or record that the beta CLI only exposes status/workspace status.
   - Launch Desktop.
   - Pass: Desktop attaches to the same Core home and owner created by CLI.

10. Concurrent Desktop/CLI launch
    - Reset to a fresh profile.
    - Start Desktop and `bin/kestrel status` at roughly the same time.
    - Pass: only one Core home, one lock owner, and one migration owner exist; the second shell attaches, waits, or reports clear readiness.

11. `kcron` beta Core-backed status/lease behavior
    - Run:
      ```sh
      ~/KestrelSmoke/cli/bin/kcron status
      ```
    - Launch a duplicate status/lease path if available.
    - Pass: output is beta-labeled, Core-backed, and duplicate lease decisions are owned by Local Core.

12. Hostile inherited `DATABASE_URL`
    - From a clean shell, run:
      ```sh
      DATABASE_URL=postgresql://host-env-should-not-be-used.invalid:5432/kestrel ~/KestrelSmoke/cli/bin/kestrel status
      ```
    - Launch Desktop from a clean profile after setting a hostile shell env where possible.
    - Pass: packaged default mode ignores the inherited host `DATABASE_URL` and still uses the Local Core managed database unless Advanced external mode is explicitly selected.

13. Stale lock/socket recovery
    - Record the live lock owner, then stop the Core owner or use the approved stale-lock fixture.
    - Leave a stale lock and, where possible, a stale API socket.
    - Launch Desktop and run CLI status.
    - Pass: stale lock/socket state is repaired or reported as a clear repair status; no split Core home or shell-private store appears.

14. Support bundle redaction and coverage
    - Generate a support bundle after Desktop and CLI have both attached.
    - Pass: key material is redacted, and the bundle includes Core home, manifest, lock, DB mode, migration status, socket presence, and legacy-state report.

## Additional Failure-Mode Probes

- Missing Postgres bundle: temporarily remove or rename only the copied artifact's bundled Postgres resource, then launch Desktop or CLI. Pass if the error is explicit and repairable.
- Migration failure: use the approved migration-failure fixture only. Pass if the migration owner and failure code are clear.
- Incompatible Core version: use a fixture lock/manifest with a higher incompatible version. Pass if Desktop/CLI report incompatible Core status instead of starting a competing Core.
- External database missing URL: select Advanced external database mode without a URL. Pass if readiness reports `LOCAL_CORE_EXTERNAL_DATABASE_URL_REQUIRED`.
- Orphaned DB process: stop the Core owner while leaving the managed database process. Pass if the next launch verifies pid/socket identity and recovers or reports repair status.

## Pass Criteria

The smoke passes only when every numbered case passes with evidence.

- Desktop, CLI, and `kcron` are shells over the same Kestrel Local Core in packaged default mode.
- There is exactly one default Core home, one Core status contract, one managed database owner, and one migration owner.
- Desktop, CLI/TUI, settings, model policy, workspaces, sessions, history, diagnostics, support bundles, kcron, and Desktop project runs do not create shell-private durable default state.
- Packaged Desktop and CLI do not depend on repo checkout, repo `.env`, host Postgres, Docker, Homebrew Postgres, or unbundled runtime resources.
- Recovery states are explicit for stale locks/sockets, orphaned database processes, missing Postgres bundle, migration failure, incompatible version, external database missing URL, and hostile inherited `DATABASE_URL`.
- Support bundles redact key material while preserving the release-debug fields listed above.

## Fail Criteria

Fail the release gate if any step creates split Core state, silently uses host `DATABASE_URL`, requires the source checkout for packaged defaults, stores durable default state under shell-private Desktop/CLI locations, hides database/migration/lock failures behind generic runtime errors, leaks key material in support bundles, or cannot be reproduced from the packaged artifacts.
