# Kestrel Desktop 0.5.1 State Bridge

Desktop `0.5.1` is an additive compatibility release. It keeps the embedded Next.js renderer and copies its allowlisted local UI state into Kestrel Local Core so the 0.6 static renderer can import that state after the repository cutover.

See also the [Desktop v0.5 clean-machine smoke checklist](./2026-06-16-desktop-v0.5-beta-clean-machine-smoke.md).

## Release Contract

- Desktop bridge protocol version `2` adds typed `getUiState` and `syncLegacyUiState` methods.
- Local Core stores a versioned `desktop-ui-state-v1` document at `settings/desktop-ui-state.json` and serves it through `GET` and `PUT /v1/desktop/ui-state`.
- The legacy renderer performs an initial snapshot, snapshots after allowlisted writes, and snapshots before page hide or backgrounding.
- Only the documented Desktop workspace, rail, theme, composer, thread, task graph, and Mission Control keys are accepted. Provider credentials and arbitrary local storage are excluded.
- A snapshot with unchanged source, app version, and entries is idempotent; it preserves the existing capture timestamp.
- Local Core accepts an existing `0.5.0-beta.0` manifest when the `0.5.1` shell starts, without rewriting the manifest version, so the previous app can still roll back after the new Core process exits. All other version mismatches remain blocked.
- The Desktop bridge does not read or overwrite TUI state in `ui-state.json`.
- Local Core authentication remains in the Electron main process. The renderer receives no Local Core token.

## Build Evidence

Run from a clean release checkout:

```bash
CI=true pnpm install --frozen-lockfile
pnpm run governance:check
pnpm run test
pnpm run prompt-suite
pnpm run evals:release-check
pnpm run desktop:package
pnpm run desktop:release-check
```

The package must report `0.5.1` in both `CFBundleShortVersionString` and `CFBundleVersion`. It must include the legacy Next.js renderer, the Local Core Desktop UI state module, and the bundled database resources. It must not include local `.env` files, private keys, or repository secrets.

## Upgrade Proof

1. Start with a real `0.5.0-beta.0` user profile containing a selected workspace, theme, at least one conversation, and Mission Control state.
2. Quit the old app and confirm no Kestrel Electron, Local Core, web renderer, or test database process remains.
3. Install `0.5.1` and launch a single supervised Electron process.
4. Confirm the legacy cockpit state is unchanged and usable.
5. Confirm `settings/desktop-ui-state.json` exists under the same Kestrel Local Core home and contains only allowlisted keys.
6. Confirm the TUI state file is byte-for-byte unchanged.
7. Quit Desktop, confirm all child processes exit, relaunch once, and verify an unchanged snapshot is idempotent.

Do not launch another Desktop instance until the current app and its child processes have exited. Every GUI proof begins and ends with a process-list check, and all isolated test state is removed before another run.

## Rollback Proof

1. Quit `0.5.1` and confirm all Kestrel processes have stopped.
2. Reinstall the previous `0.5.0-beta.0` artifact against the same user profile.
3. Confirm the previous Desktop still reads its original local storage and ignores the additive Local Core bridge file.
4. Reinstall `0.5.1`, launch one supervised process, and verify the same allowlisted snapshot is recovered without duplicate or rewritten TUI state.

Rollback does not delete `desktop-ui-state.json`; the previous app ignores it. Do not promote the 0.6 static renderer until both upgrade and rollback evidence have been recorded against a real user profile.

## Operator Record

Record the source commit, artifact SHA-256, macOS version, previous artifact version, Local Core home path class (default or isolated), before/after process lists, bridge file key list, TUI state hash, launch result, shutdown result, and cleanup result. Never record provider key values or Local Core tokens.

## Existing Profile Upgrade And Rollback Evidence

The packaged `0.5.1` app and recovery `0.5.0-beta.0` app passed the state-preservation gate on 2026-07-10 against an isolated copy of the existing Electron browser profile:

- Recovery companion source commit: `aa118494e60283cef0a15f0eba0d7d1102552dca`; artifact SHA-256: `48a7bba44636a09ff2eaa8bf487a11ed477fccfe6aa03ae06a1685ad99b12f38`.
- The fixture contained the real allowlisted composer draft, prompt history, theme, task graph, thread store, active thread, Desktop workspace v4/v5, and rail state. Evidence records key names, byte lengths, and SHA-256 values only; it does not record state values.
- The Core fixture started with a `0.5.0-beta.0` manifest matching the existing product profile.
- Packaged `0.5.1` reached `ready` with a healthy managed database and bridge protocol version `2`.
- All nine copied local-storage entries matched the persisted `desktop-ui-state-v1` entries byte-for-byte by length and SHA-256.
- The Core manifest remained `0.5.0-beta.0`, preserving the version contract needed by the previous shell after shutdown.
- App, Local Core, Postgres, renderer, and helper processes all exited before the probe returned.
- The full sequence was recovery `0.5.0-beta.0`, packaged `0.5.1`, then recovery `0.5.0-beta.0`; every launch reached `ready` with a healthy database and bridge versions `1 -> 2 -> 1`.
- Eight non-task-graph keys were byte-identical across all three launches. The task graph was structurally identical; its 79 differences were exclusively valid ISO `updatedAt` values refreshed by normal renderer startup.
- The rollback app ignored the retained `desktop-ui-state-v1` file and continued reading legacy browser state.
- The live TUI state hash remained `729a14686b9a65dd2683bba54daefe65dd9166a2c3f9646a4fc0df10861b1a12` before and after.

The original archived `0.5.0-beta.0` zip remains unsuitable for rollback because it lacks a complete signature, packaged TypeScript, and a production `.next` renderer. The recovery artifact repairs only packaging and process safety while retaining version `0.5.0-beta.0`, bridge protocol `1`, and the old renderer behavior.
