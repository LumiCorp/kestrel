The repo already names its validation gates clearly. The right improvement is to use them consistently, not to invent a new ad hoc check every time a feature changes.

## Core gates

```bash
pnpm run governance:check
pnpm run test
pnpm run prompt-suite
```

For runtime and determinism work:

```bash
pnpm run evals:release-check
```

## What each gate protects

| Gate | Protects |
| --- | --- |
| `governance:check` | structural rules and contract discipline |
| `test` | code-level behavior |
| `prompt-suite` | reference-agent prompt behavior |
| `evals:release-check` | behavior ownership and released Ruhroh contract validation |

## Operator consequence

When a nightly report goes missing or a streamed route starts behaving differently, these gates are how you distinguish:

- product prompt drift
- runtime regression
- structural contract breakage
- replay instability

Skipping them turns every incident into open-ended guesswork.
