Operating Kestrel means more than keeping a process online. You should be able to identify the affected session and run, see what happened, choose a safe action, and verify that the same user path works afterward.

## Prepare and deploy

Start with [the runner service](/operate/runner-service), then configure [environment and authentication](/operate/environment-and-auth) and review the [production operating model](/operate/deployment). These guides keep runner and provider credentials on trusted servers and establish the health checks required before serving users.

## Investigate and recover

Use [Observability](/operate/observability) to locate the request, [Reliability](/operate/reliability) to structure the incident, [Replay](/operate/replay) to inspect persisted evidence, and [Troubleshooting](/operate/troubleshooting) to start from a visible symptom.

## Control and quality

[Operator control](/operate/operator-control) acts on work already in flight. [Ruhroh evaluations](/operate/evaluations) and [quality gates](/operate/quality-gates) help compare behavior before and after a change. Kestrel One model operators should also understand [credential leases](/operate/credential-leases) and [model access decisions](/operate/model-authority).
