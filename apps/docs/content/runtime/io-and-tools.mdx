Kestrel reaches models and external systems through explicit gateways. A model
can request a tool, but the Runtime validates the tool input and applies the
configured capability and approval policy before the effect occurs.

## Tool flow

```text
run step
  -> model requests a named tool
    -> Runtime validates its structured input
      -> capability and approval policy are applied
        -> filesystem, shell, network, provider, or MCP operation runs
          -> structured result and events are recorded
```

This boundary makes it possible to distinguish a rejected request, a tool
failure, a waiting approval, and a Runtime failure. Applications do not need to
infer those conditions from model prose.

## Source changes

Source reads identify the bytes and revision observed by the model. Partial
reads remain visibly partial. Edits to an existing file are checked against the
observed revision so a stale proposal does not silently overwrite newer work.

Tool output, patch artifacts, and workspace checkpoints provide inspectable
evidence of the requested and applied effects. A checkpoint supports recovery
and comparison; it does not by itself claim that the change was correct.
