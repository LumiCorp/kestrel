The Runtime owns the lifecycle of runs and steps. Applications submit work and
display its state; they do not implement their own definitions of started,
waiting, completed, failed, or cancelled.

## Execution flow

```text
product or server application
  -> Local Core or runner service
    -> Execution Protocol
      -> Runtime starts or continues a run
        -> model and tool steps execute
          -> events and checkpoints persist
            -> terminal result is recorded
```

This is the same flow for Desktop, the CLI and TUI, Kestrel One, and SDK
applications. The deployment target changes; the run lifecycle does not.

## What to inspect

When execution looks wrong, start with the durable identities and states:

1. confirm that a run started
2. identify its session and thread
3. inspect its ordered events and tool outcomes
4. read the completed, waiting, failed, or cancelled terminal event
5. compare that result with what the calling product displayed

A disconnected stream does not determine the run's outcome. The recorded
terminal event does.
