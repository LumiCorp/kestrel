The runner service gives your application one HTTP endpoint for starting and observing Kestrel work. Your browser continues to call your application, and your application keeps the runner credential private.

## Keep the application and runtime separate

This arrangement keeps three responsibilities clear:

- the browser only speaks to the app
- the app server owns user auth and service credentials
- the runtime owns durable execution, storage, and replay evidence

Do not call the runner directly from browser code. Doing so would expose the runner credential and allow browser input to define trusted user or tenant information.

## Start the service locally

```bash
kestrel web
```

That command starts the runner service and prints copy-paste-ready values for:

- `KESTREL_RUNNER_SERVICE_URL`
- `KESTREL_RUNNER_SERVICE_TOKEN`

## Connect Workspace Copilot

```bash
export KESTREL_RUNNER_SERVICE_URL=http://127.0.0.1:4010
export KESTREL_RUNNER_SERVICE_TOKEN=dev-secret
pnpm dev
```

With those values in place, the shared `workspaceCopilotAgent` can serve:

- `/api/copilot/run`
- `/api/copilot/stream`
- `/api/copilot/webhook`

## Request flow

```text
Browser
  -> POST /api/copilot/stream
    -> Next.js server route
      -> workspaceCopilotAgent.stream()
        -> runner service
          -> runtime
            -> store/events/tools/model
```

## Check runner health

Use these quick checks before you debug the application routes:

```bash
echo "$KESTREL_RUNNER_SERVICE_URL"
echo "$KESTREL_RUNNER_SERVICE_TOKEN"
curl -I "$KESTREL_RUNNER_SERVICE_URL/health"
```

The health check should succeed before you spend time on route-level debugging. A healthy app with an unhealthy runner is still a broken Kestrel deployment.

## Send a test command

After the health check succeeds, verify both the OpenAI-compatible endpoint and the command endpoint:

```bash
curl -sS "$KESTREL_RUNNER_SERVICE_URL/v1/models" \
  -H "Authorization: Bearer $KESTREL_RUNNER_SERVICE_TOKEN"

curl -sS "$KESTREL_RUNNER_SERVICE_URL/commands" \
  -H "Authorization: Bearer $KESTREL_RUNNER_SERVICE_TOKEN" \
  -H "Content-Type: application/json" \
  -H "x-kestrel-actor-id: user-123" \
  -H "x-kestrel-actor-type: end_user" \
  -H "x-kestrel-actor-name: Taylor Example" \
  -H "x-kestrel-tenant-id: acme" \
  -d '{
    "command": "runner.ping"
  }'
```

Example `runner.ping` response:

```json
{
  "type": "runner.pong",
  "status": "ok"
}
```

If `/health` works but `/commands` fails, check the token and actor headers. If `/commands` works but `/api/copilot/run` fails, inspect the application route and its environment settings.

## Check the local request path

- Workspace Copilot now has a live runner
- the Next.js routes can proxy browser requests safely
- the browser still never sees runner credentials
- health, command, and application-route failures can be checked separately

## Failure mode: runner unavailable

If the runner is down:

- `/api/copilot/run` fails even though the app is healthy
- `/api/copilot/stream` can close early or emit an upstream error
- `kcron` cannot launch the nightly review job successfully

Monitor the runner as you would any service that your application depends on.

## What to inspect first when the app looks broken

Use this order:

1. confirm `/health`
2. confirm `GET /v1/models`
3. confirm `/commands` with `runner.ping`
4. only then inspect `/api/copilot/*` routes

This order helps you find whether the problem begins in the runner, its credentials, or the application route.

## Minimal local operator loop

Use this sequence to check the complete local request path before debugging application behavior:

```bash
kestrel web
export KESTREL_RUNNER_SERVICE_URL=http://127.0.0.1:4010
export KESTREL_RUNNER_SERVICE_TOKEN=dev-secret
pnpm dev
curl -I "$KESTREL_RUNNER_SERVICE_URL/health"
curl -sS "$KESTREL_RUNNER_SERVICE_URL/v1/models" \
  -H "Authorization: Bearer $KESTREL_RUNNER_SERVICE_TOKEN"
```
