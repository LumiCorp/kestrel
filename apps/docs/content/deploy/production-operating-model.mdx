Production Kestrel is not “the service is reachable.” The operating model assumes that runs are worth inspecting, recurring work should stay explicit, and quality gates belong next to the code that changes runtime behavior.

## Core production shape

| Layer | Responsibility |
| --- | --- |
| application server | user auth, actor context, route exposure |
| runner service | execution boundary and service auth |
| runtime/store | durable state, event history, replay evidence |
| workspace | prompts, automation contract, output artifacts |
| operations | evaluations, quality gates, incident handling |

## Validate the operating model

```bash
pnpm run governance:check
pnpm run test
pnpm run prompt-suite
```

If you changed runtime or determinism behavior:

```bash
pnpm run evals:release-check
```

## What changes in this step

- Workspace Copilot is now treated like a real operated product
- nightly `kcron` work becomes an explicit operational surface
- replay, traces, reports, and released evaluations become part of supportability

## What this means for Workspace Copilot

When the nightly review starts drifting, the failure is not only “a cron job failed.” It can mean:

- the workspace prompt no longer matches the intended output
- memory continuity is stale or conflicting
- the stream route and scheduler path no longer produce comparable results
- a runtime or tool contract changed under the example

That is why Kestrel ties deployment directly to reliability, evaluations, and replay.

## Operator checklist

- [ ] application routes still use server-owned context resolution
- [ ] runner credentials stay out of client bundles
- [ ] workspace catalog and automation eligibility are explicit
- [ ] report artifacts are written into `./reports`
- [ ] traces and replay evidence are inspectable after failures
- [ ] repo validation gates run before shipping behavioral changes
