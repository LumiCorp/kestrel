Environment and auth are where many integrations accidentally turn a sound local demo into an unsafe production shape. The Kestrel rule is simple: the application server owns trust, and the runner service receives structured actor context plus service authentication.

## Core environment values

These values stay on the server:

```bash
export KESTREL_RUNNER_SERVICE_URL=http://127.0.0.1:4010
export KESTREL_RUNNER_SERVICE_TOKEN=dev-secret
```

Do not expose the token through browser-readable environment variables.

## Resolve the demo context in one place

```ts
export async function resolveWorkspaceCopilotContext() {
  return {
    actor: {
      actorId: "user-123",
      actorType: "end_user" as const,
      displayName: "Taylor Example",
    },
    tenantId: "acme",
  };
}
```

Keeping this logic in one server module makes the product contract auditable.

## Correlation headers

The Next.js route helpers read these headers from the inbound request:

- `x-request-id`
- `x-correlation-id`
- `user-agent`
- `x-forwarded-for`

If ids are missing, the helper generates them and returns Kestrel correlation ids in the response headers. That matters when operators later need to line up a browser request with a runner trace or failure.

## Environment and request ownership

| Concern | Owned by |
| --- | --- |
| user authentication | application server |
| runner bearer token | application server |
| actor id / actor type / tenant id | application server |
| run lifecycle and persistence | runner service + runtime |

## What changes in this step

- the demo now has a stable server-side auth and environment contract
- actor and tenant context no longer live as ad hoc inline examples
- `/api/copilot/run` and `/api/copilot/stream` preserve request correlation end to end

## Failure mode: context drift

If requests fail because actor or tenant context is missing, debug the server route, not the browser payload. The browser should not be trusted to define `actorId` or `tenantId` for the canonical example.
