Workspace Copilot is a practical example of an assistant that stays connected to one project. It answers questions while you work, remembers useful findings and decisions, and can produce a scheduled review of the same workspace each night.

The result is one assistant that can support:

- interactive requests from an application route
- remembered findings and decisions between runs
- traces you can inspect after work finishes
- recurring reviews tied to visible workspace settings

## Topology

```text
Browser
  -> Next.js route handlers
    -> runner service
      -> Kestrel runtime
        -> store, tools, model gateway, event history
          -> workspace reports and traces
```

The browser calls your application routes, while the runner token remains on the application server.

## File layout

The complete setup combines a small application with settings stored alongside the workspace:

| Path | Purpose |
| --- | --- |
| `lib/workspace-copilot.ts` | agent definition, shared actor context, tracer wrapper |
| `app/api/copilot/run/route.ts` | synchronous JSON route |
| `app/api/copilot/stream/route.ts` | SSE streaming route |
| `app/api/copilot/webhook/route.ts` | webhook/background entrypoint |
| `global profile selection` | workspace-specific model, prompt, output, and schedule settings |
| `selected global profile` | baseline system guidance |
| `interactive prompt` | interactive run prompt |
| `central scheduler prompt` | nightly automation prompt |
| `reports/nightly-review.md` | expected recurring report artifact |

## `~/.kestrel/workspaces.json`

The workspace id should be stable and machine-local. That is what lets the catalog, sessions, and future scheduler agree on the same folder identity.

```json
{
  "version": 3,
  "workspaces": [
    {
      "id": "local:example",
      "rootPath": "/path/to/workspace-copilot",
      "label": "workspace-copilot",
      "automationEnabled": false
    }
  ]
}
```

## `lib/workspace-copilot.ts`

This shared server-side module gives every route the same agent and user context.

```ts
import { createAgent } from "@kestrel-agents/sdk";
import { createTracer, InMemoryTraceProcessor } from "@kestrel-agents/observability";

export const workspaceCopilotAgent = createAgent({
  id: "workspace-copilot",
  profileId: "workspace-copilot",
  target: {
    kind: "remote",
    baseUrl: process.env.KESTREL_RUNNER_SERVICE_URL!,
    authToken: process.env.KESTREL_RUNNER_SERVICE_TOKEN!,
  },
});

export const workspaceCopilotContext = {
  actor: {
    actorId: "user-123",
    actorType: "end_user" as const,
    displayName: "Taylor Example",
  },
  tenantId: "acme",
};

const tracer = createTracer({
  processors: [new InMemoryTraceProcessor()],
});

export const tracedWorkspaceCopilotAgent = tracer.wrapAgent(workspaceCopilotAgent);
```

Reusing the module prevents different routes from accidentally identifying the same agent or user in different ways.

## `app/api/copilot/run/route.ts`

Use the JSON route for standard request-response turns:

```ts
import { createJsonRunRouteHandler } from "@kestrel-agents/next";

import { tracedWorkspaceCopilotAgent, workspaceCopilotContext } from "@/lib/workspace-copilot";

export const POST = createJsonRunRouteHandler({
  agent: tracedWorkspaceCopilotAgent,
  resolveContext: async () => workspaceCopilotContext,
});
```

## `app/api/copilot/webhook/route.ts`

Use the webhook route for server-driven or background entrypoints that map an external payload into a Kestrel turn:

```ts
import { createWebhookRunRouteHandler } from "@kestrel-agents/next";

import { tracedWorkspaceCopilotAgent, workspaceCopilotContext } from "@/lib/workspace-copilot";

export const POST = createWebhookRunRouteHandler<{ sessionId?: string; prompt: string }>({
  agent: tracedWorkspaceCopilotAgent,
  resolveContext: async () => workspaceCopilotContext,
  mapPayload(payload) {
    return {
      sessionId: payload.sessionId ?? "workspace-copilot-nightly-001",
      message: payload.prompt,
    };
  },
});
```

## `global profile selection`

This file keeps the model, prompts, output folder, and schedule with the workspace. Those settings do not silently merge with machine-wide profiles, so another person can review what the workspace expects before running it.

```json
{
  "version": 1,
  "label": "Workspace Copilot",
  "agent": "workspace-copilot",
  "sessionPrefix": "workspace-copilot",
  "modelProvider": "openai",
  "model": "gpt-5.4-2026-03-05",
  "promptFiles": ["./selected global profile"],
  "outputRoot": "./reports",
  "cron": {
    "enabled": true,
    "schedule": "0 22 * * *",
    "launchPrompt": "./central scheduler prompt"
  }
}
```

## Prompt files

The system prompt keeps the assistant grounded in the workspace:

```md
You are Workspace Copilot for this repository.

Read the workspace before making claims. Keep outputs concrete, cite files when relevant, and write durable artifacts into ./reports when the prompt asks for them.
```

An interactive prompt can stay lightweight:

```md
Review the current workspace changes. Summarize the highest-risk files, note missing tests, and propose the next review actions.
```

The nightly prompt should define a durable artifact:

```md
Review the workspace changes since the previous run. Write a report to ./reports/nightly-review.md with sections for:

1. Significant changes
2. Risks and open questions
3. Missing tests or validation
4. Next actions for tomorrow
```

## Memory bootstrap files

The workspace can start with small plain-text files that make its initial context easy to review:

```md
# workspace catalog entry

The workspace currently prioritizes docs, replay, and operator-control polish.
```

```md
# catalog metadata

- Reports must be written into ./reports.
- Nightly review should call out operator impact before low-level cleanup.
```

```md
# session history

If the nightly run cannot finish, leave the next operator a clear summary of blockers and the expected follow-up route.
```

## Session memory

Session memory carries findings and decisions from one run to the next:

```ts
const session = workspaceCopilotAgent.session("workspace-copilot-session-001");
const memory = await session.memory.get(workspaceCopilotContext);

await session.memory.update(
  {
    expectedRevision: memory.revision,
    patch: {
      findings: "Docs and runtime pages changed in the last review cycle.",
      decisions: "Nightly review should call out operator impact first.",
      nextAction: "Write the update to reports/nightly-review.md.",
      linkedArtifacts: ["reports/nightly-review.md"],
    },
  },
  workspaceCopilotContext,
);
```

This is what lets an interactive daytime session and a nightly run share continuity without pretending they are the same long-lived request.

## Example output artifact

The nightly automation writes a report that anyone with project access can inspect:

```md
# Nightly Review

## Significant changes
- Expanded Workspace Copilot docs with full route examples.
- Added operator guidance for runner troubleshooting and replay inspection.

## Risks and open questions
- Streaming route behavior still needs manual verification after deploy.
- The nightly review prompt should stay aligned with current output expectations.

## Missing tests or validation
- Add docs assertions for report artifact references.
- Confirm search finds session-memory content after the rewrite.

## Next actions for tomorrow
- Run docs build and governance gates.
- Review the generated report with the owning team.
```

## Route and webhook request examples

Interactive JSON route:

```bash
curl -sS \
  -X POST http://127.0.0.1:3000/api/copilot/run \
  -H 'content-type: application/json' \
  -d '{
    "sessionId": "workspace-copilot-session-001",
    "message": "Summarize the riskiest docs changes and list follow-up tasks."
  }'
```

Streaming route:

```bash
curl -N \
  -X POST http://127.0.0.1:3000/api/copilot/stream \
  -H 'content-type: application/json' \
  -d '{
    "sessionId": "workspace-copilot-session-001",
    "message": "Stream the review as you inspect the workspace."
  }'
```

Webhook route:

```bash
curl -sS \
  -X POST http://127.0.0.1:3000/api/copilot/webhook \
  -H 'content-type: application/json' \
  -d '{
    "sessionId": "workspace-copilot-nightly-001",
    "prompt": "Review the workspace and write ./reports/nightly-review.md."
  }'
```

## Run Workspace Copilot end to end

Use this sequence locally:

```bash
kestrel web
export KESTREL_RUNNER_SERVICE_URL=http://127.0.0.1:4010
export KESTREL_RUNNER_SERVICE_TOKEN=dev-secret
pnpm dev
kestrel workspace status
kestrel workspace list
kcron install
kcron run-once
```

## What this gives you

Workspace Copilot shows how to turn a useful interactive assistant into something your team can return to and operate over time:

- expose safe application routes
- identify the person and organization behind each request
- persist continuity through versioned memory
- bind recurring work to the workspace instead of a hidden global schedule
- leave behind enough traces, reports, and replay evidence to support operators later
