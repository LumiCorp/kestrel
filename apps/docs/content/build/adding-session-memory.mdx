Session memory gives your application a readable record of what should carry forward. Each update has a revision number, so concurrent changes cannot silently overwrite one another.

For **Workspace Copilot**, memory is the bridge between a daytime review and a nightly automated report.

## Connect interactive and scheduled work

Workspace Copilot does work in two ways:

- an interactive daytime request from `/api/copilot/run` or `/api/copilot/stream`
- a nightly review launched by `kcron`

Session memory lets both kinds of work share useful context without keeping one browser request open indefinitely.

## Read the current memory snapshot

```ts
const session = workspaceCopilotAgent.session("workspace-copilot-session-001");
const memory = await session.memory.get(workspaceCopilotContext);
```

The returned value includes a revision number and a structured snapshot. That revision is what protects the update path from silent overwrites.

## Update memory with a versioned patch

```ts
await session.memory.update(
  {
    expectedRevision: memory.revision,
    patch: {
      findings: "Docs and runtime pages changed in the last review cycle.",
      decisions: "Nightly review should focus on operator impact first.",
      openQuestions: "Do the stream and webhook routes still return the same correlation headers?",
      nextAction: "Write the nightly report to ./reports/nightly-review.md.",
      linkedArtifacts: ["reports/nightly-review.md"],
    },
  },
  workspaceCopilotContext,
);
```

## Memory fields that matter

| Field | What it remembers |
| --- | --- |
| `findings` | captures the current state of the workspace |
| `decisions` | records choices a later run should respect |
| `openQuestions` | carries unresolved questions into later work |
| `nextAction` | keeps the next run anchored in real work |
| `linkedArtifacts` | points to project outputs like `reports/nightly-review.md` |

## How the memory path works

```text
session()
  -> read the current memory snapshot
    -> apply versioned patch
      -> persist next revision
        -> a later run or person can inspect it
```

## Resolve a revision conflict

If two processes update the same session at once, the revision check rejects the later write instead of silently overwriting newer information. Read the latest revision, review the competing change, and apply the intended update again.

<Outcome>
  <p>A later run can read the new memory revision and its linked report without reconstructing continuity from prompt text.</p>
</Outcome>

## Use the saved memory

- Workspace Copilot now has readable, versioned memory
- later nightly runs can reuse saved context instead of relying only on prompt text
- people can inspect and correct what the next run will remember
- `linkedArtifacts` connects the memory record to `reports/nightly-review.md`

## Investigate an incorrect result

If a nightly report is wrong, inspect:

- what memory existed before the run
- which revision was current
- which file the previous run intended to update

The memory record gives you a concrete place to understand and correct continuity when a later result is wrong.
