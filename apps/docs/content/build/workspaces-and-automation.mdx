Kestrel now treats workspaces as entries in the machine-local catalog at `~/.kestrel/workspaces.json`.
Starting Kestrel from a project registers the project root in that catalog. It does not write `.kestrel` files into the project.
In the Workspace Copilot path, that catalog entry is the stable identity for the local project.

## Inspect the catalog

```bash
cd /path/to/project
kestrel workspace status
kestrel workspace list
```

When the current directory is inside a Git repository, the Git root becomes the workspace root.
For non-Git folders, Kestrel registers the current directory.

## What the catalog stores

| Field | Purpose |
| --- | --- |
| workspace id | stable local identity derived from the root |
| root path | project or folder root |
| launch cwd | original launch directory when different from the root |
| label | display name |
| automationEnabled | future scheduler eligibility flag |

Profiles stay global. Runtime behavior comes from the selected profile and the active session, not from project-local workspace files.

## Automation status

`automationEnabled` is reserved for the central scheduler.
Workspace-backed `kcron` execution reads catalog entries, not project-local runtime files.

The next scheduler pass should build directly on the catalog shape.

Use `kestrel workspace status` and `kestrel workspace list` for catalog inspection.
