Streaming is the first place many integrations blur request and session semantics. Kestrel keeps them separate. A stream belongs to one run. Session continuity still belongs to the session id.

For **Workspace Copilot**, this is the first live-response path in the tutorial.

## Why this exists

Use `stream()` when the application needs live progress or incremental output. Do not use it as a substitute for background monitoring or session history.

That distinction matters later because the same example will also add:

- background subscriptions for non-request-bound observation
- explicit session memory for continuity
- scheduled workspace automation through `kcron`

## Start the stream

```ts
const stream = workspaceCopilotAgent.stream(
  {
    sessionId: "workspace-copilot-session-001",
    message: "Walk me through the risky files in this workspace and stream progress as you go.",
  },
  workspaceCopilotContext,
);
```

The call returns a `RunnerStream`. It is both an async iterable of events and a promise-like result holder for the terminal event.

## Consume request-scoped events

```ts
for await (const event of stream) {
  console.log(event.type, event.payload);
}

const terminal = await stream.result;
console.log(terminal.type);
```

That shape is important:

| Part | Meaning |
| --- | --- |
| `for await ... of stream` | live events for one run |
| `await stream.result` | terminal outcome for that same run |
| `stream.cancel()` | explicit cancellation of that same run |

## Request lifecycle

```text
app code
  -> workspaceCopilotAgent.stream()
    -> runner service
      -> runtime emits step/run events
        -> app forwards events to the caller
          -> terminal event resolves stream.result
```

## Handle cancellation intentionally

If the caller disconnects or explicitly cancels:

```ts
const stream = workspaceCopilotAgent.stream(
  {
    sessionId: "workspace-copilot-session-001",
    message: "Explain the deployment status and stop if I close the page.",
    signal: abortController.signal,
  },
  workspaceCopilotContext,
);
```

The SDK cancels that exact upstream run and `stream.result` resolves with `run.cancelled`. That becomes important once this stream is exposed through `/api/copilot/stream`.

## What changes in this step

- the example now supports request-scoped live output
- the app can later map this stream to an SSE route without changing the agent contract
- session continuity still belongs to `workspace-copilot-session-001`
- no workspace scheduler or background watcher has been introduced yet

## What this does not solve

Streaming does not replace:

- session memory
- replayable run history
- background subscriptions
- recurring automation

Those are separate concerns. Kestrel keeps them separate so failures stay explainable.
