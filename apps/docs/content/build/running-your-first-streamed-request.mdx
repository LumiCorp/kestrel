Turn the first Workspace Copilot request into a stream so your application can show activity before the complete answer is ready.
</WorkspaceCopilotDemo>

Some tasks take long enough that a silent loading state leaves users wondering whether anything is happening. Streaming lets your application display progress as Kestrel works, while still providing one final result when the run ends.

## When to stream

Use `stream()` when someone is waiting for the current request and benefits from seeing progress or incremental output. For work that continues after the original request ends, use a [background subscription](/build/adding-background-subscriptions) instead.

## Start the stream

```ts
const stream = workspaceCopilotAgent.stream(
  {
    sessionId: "workspace-copilot-session-001",
    message: "Walk me through the risky files in this workspace and stream progress as you go.",
  },
  workspaceCopilotContext,
);
```

The call returns a `RunnerStream`. Read it in two ways: iterate over live events as they arrive, then await `stream.result` for the final outcome.

## Read progress and the final result

```ts
for await (const event of stream) {
  console.log(event.type, event.payload);
}

const terminal = await stream.result;
console.log(terminal.type);
```

Each part has one job:

| Part | Meaning |
| --- | --- |
| `for await ... of stream` | live events for one run |
| `await stream.result` | terminal outcome for that same run |
| `stream.cancel()` | cancel that run when the caller no longer wants it |

## Request lifecycle

```text
app code
  -> workspaceCopilotAgent.stream()
    -> runner service
      -> runtime emits step/run events
        -> app forwards events to the caller
          -> terminal event resolves stream.result
```

## Cancel work when the caller leaves

If the caller disconnects or explicitly cancels:

```ts
const stream = workspaceCopilotAgent.stream(
  {
    sessionId: "workspace-copilot-session-001",
    message: "Explain the deployment status and stop if I close the page.",
    signal: abortController.signal,
  },
  workspaceCopilotContext,
);
```

When the signal is aborted, the SDK cancels the run and `stream.result` resolves with `run.cancelled`. Your application can now distinguish a deliberate cancellation from a failure or a completed response.

## What works now

- your application can show progress while Workspace Copilot works
- the final result still provides a clear completed, waiting, failed, or cancelled outcome
- reusing `workspace-copilot-session-001` keeps the next request connected to this one
- the same stream can be exposed through a Next.js SSE route when you add a browser UI
