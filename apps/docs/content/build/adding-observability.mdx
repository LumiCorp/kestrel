Tracing helps you answer what happened after a request has finished, especially when the result is missing, late, or unexpected.

## Decide what your traces should answer

Once Workspace Copilot streams, remembers context, and runs on a schedule, you need to answer practical questions:

- did the run fail, cancel, or complete?
- which session and thread did it touch?
- which route or integration started it?
- which actor and tenant were attached?

The observability package records those details consistently across runs, streams, resumes, and subscriptions.

## Install the package

```bash
pnpm add @kestrel-agents/observability@0.6.0-beta.0 @kestrel-agents/sdk@0.6.0-beta.0
```

## Create a tracer

```ts
import { createTracer, InMemoryTraceProcessor } from "@kestrel-agents/observability";

const tracer = createTracer({
  processors: [new InMemoryTraceProcessor()],
});
```

## Wrap the agent

```ts
import { workspaceCopilotAgent } from "@/lib/workspace-copilot";

export const tracedWorkspaceCopilotAgent = tracer.wrapAgent(workspaceCopilotAgent);
```

That wrapper instruments:

| Method | Span kind |
| --- | --- |
| `run()` | `run` |
| `stream()` | `stream` |
| `resume()` | `resume` |
| `subscribe()` | `subscription` |

## Use the traced agent in a route

```ts
export const POST = createStreamRunRouteHandler({
  agent: tracedWorkspaceCopilotAgent,
  resolveContext: async () => workspaceCopilotContext,
});
```

The route uses the wrapped agent exactly as it used the original one.

## OTEL export bridge

Kestrel keeps the primary trace model Kestrel-native, then bridges outward when you need vendor-neutral export:

```ts
import { OpenTelemetryTraceExporter } from "@kestrel-agents/observability";

const exporter = new OpenTelemetryTraceExporter();
```

Your application continues to work with Kestrel runs, streams, and subscriptions while the exporter translates those traces for OpenTelemetry tools.

<Outcome>
  <p>A completed, failed, cancelled, or waiting run can be located by session and run identity without reconstructing the story from unrelated console messages.</p>
</Outcome>

## What tracing now covers

- `workspace-copilot` now emits traces for `run`, `stream`, `resume`, and `subscribe`
- the same trace model can cover daytime requests and nightly `kcron` runs
- you can investigate failures without assembling the story from unrelated logs

## Investigate a missing report

When a nightly report is missing, a trace should help you answer whether the failure lived in:

- route entry
- runner communication
- runtime execution
- cancellation
- downstream export or processing

That information tells you where to investigate instead of searching unrelated logs.
