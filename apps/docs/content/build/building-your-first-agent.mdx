<WorkspaceCopilotDemo step="Step 1">
Build a small Workspace Copilot that reviews a project and returns a useful answer you can show in your application.
</WorkspaceCopilotDemo>

By the end of this tutorial, you will have a TypeScript script that asks Kestrel to review a workspace. The response includes readable text for your user and the identifiers your application needs to continue the same session later.

You do not need a web framework for this first step. Starting with a script makes it easier to see the request, the response, and the few values that connect them.

## Before you start

You need Node.js 20 or newer and a running Kestrel runner service. Start your local runner with `kestrel web`; the command prints the URL and token you will use below.

You also need a runner profile named `workspace-copilot`. A profile tells Kestrel which model, tools, and working environment this agent may use.

## 1. Install the SDK

Install the Kestrel SDK and a TypeScript runner for the script:

```bash
pnpm add @kestrel-agents/sdk@0.6.0-beta.0
pnpm add -D tsx
```

## 2. Add your local connection settings

Copy the values printed by `kestrel web` into your terminal. A typical local setup looks like this:

```bash
export KESTREL_RUNNER_SERVICE_URL=http://127.0.0.1:4010
export KESTREL_RUNNER_SERVICE_TOKEN=dev-secret
```

Keep the token on your server. It allows code to submit work to your runner, so it should never be included in browser JavaScript or accepted from an end user.

## 3. Create the agent

Create `lib/workspace-copilot.ts`:

```ts
import { createAgent } from "@kestrel-agents/sdk";

export const workspaceCopilotAgent = createAgent({
  id: "workspace-copilot",
  profileId: "workspace-copilot",
  target: {
    kind: "remote",
    baseUrl: process.env.KESTREL_RUNNER_SERVICE_URL!,
    authToken: process.env.KESTREL_RUNNER_SERVICE_TOKEN!,
  },
});

export const workspaceCopilotContext = {
  actor: {
    actorId: "user-123",
    actorType: "end_user" as const,
    displayName: "Taylor Example",
  },
  tenantId: "acme",
};
```

The agent definition gives this assistant a name inside your application and connects it to the `workspace-copilot` runner profile. The context identifies who is asking and which organization they belong to. Kestrel records that information with the run so later activity can be attributed to the right person and tenant.

You can use these values for the first request. In an application, replace `user-123` and `acme` with the identity and organization from your authenticated server session.

## 4. Ask Kestrel to review the workspace

Create `scripts/workspace-copilot-smoke.ts`. This script sends a concrete task and prints the complete response so you can see what Kestrel gives your application:

```ts
import {
  workspaceCopilotAgent,
  workspaceCopilotContext,
} from "../lib/workspace-copilot.js";

const terminal = await workspaceCopilotAgent.run(
  {
    sessionId: "workspace-copilot-session-001",
    message: "Summarize the changes in this workspace and list the next review tasks.",
  },
  workspaceCopilotContext,
);

console.log(terminal.type);
```

This is enough to prove the runner boundary is wired correctly.

## 4a. Full smoke-test file

Put the setup and the first run into one server-only script so the initial integration is copy-pasteable:

```ts
import "server-only";

import { createAgent } from "@kestrel-agents/sdk";

const workspaceCopilotAgent = createAgent({
  id: "workspace-copilot",
  profileId: "workspace-copilot",
  target: {
    kind: "remote",
    baseUrl: process.env.KESTREL_RUNNER_SERVICE_URL!,
    authToken: process.env.KESTREL_RUNNER_SERVICE_TOKEN!,
  },
});

const context = {
  actor: {
    actorId: "user-123",
    actorType: "end_user" as const,
    displayName: "Taylor Example",
  },
  tenantId: "acme",
};

const terminal = await workspaceCopilotAgent.run(
  {
    sessionId: "workspace-copilot-session-001",
    message: "Summarize the changes in this workspace and list the next review tasks.",
  },
  context,
);
console.log(JSON.stringify(terminal, null, 2));
```

Run it from the root of your project:

```bash
pnpm exec tsx scripts/workspace-copilot-smoke.ts
```

Kestrel uses `workspace-copilot-session-001` to keep this request connected to future requests. Reuse that session id when you want the agent to continue the same piece of work; choose a new one when you want to start a separate conversation.

<Callout title="Why there is no server-only import">
  <p><code>server-only</code> is an optional Next.js guard that catches accidental imports into Client Components. This standalone script does not need it, and the Kestrel SDK does not add that build-time warning for you. Keep the SDK and runner token in scripts, server routes, or other server-owned modules.</p>
</Callout>

## 5. Read the result

A successful response has this general shape. The assistant's wording will change with the contents of your workspace.

```json
{
  "type": "run.completed",
  "runId": "run_01workspacecopilot",
  "sessionId": "workspace-copilot-session-001",
  "payload": {
    "result": {
      "assistantText": "The highest-risk changes are in the runner route and session persistence. Review those files first.",
      "output": {
        "status": "COMPLETED",
        "sessionId": "workspace-copilot-session-001",
        "runId": "run_01workspacecopilot",
        "errors": []
      }
    }
  }
}
```

The fields serve different parts of your application:

| Field | What it gives you |
| --- | --- |
| `payload.result.assistantText` | The answer you can present to the user |
| `type` | Whether the run completed, paused, failed, or was cancelled |
| `sessionId` | The conversation to reuse for a follow-up request |
| `runId` | The individual run to find in logs and observability tools |
| `payload.result.output` | Structured result data for application logic |

If the response is not `run.completed`, keep the output: it usually contains the information you need to find the problem. First confirm that the runner is still running, the URL and token match the values it printed, and the `workspace-copilot` profile exists. Then use [Troubleshooting](/operate/troubleshooting) to work from the response you received. A `run.waiting` response is different from a failure—it means the agent needs input or approval before it can continue.
