Scheduled work stays attached to the workspace that owns its prompt, settings, and output.

## Make the task repeatable

Good recurring automation is:

- local to the workspace that owns the work
- explicit about what artifact it writes
- safe to run as a fresh session
- inspectable after the fact

That is why `Workspace Copilot` writes reports into `./reports` and keeps continuity in explicit memory rather than relying on one immortal session.

## Nightly workspace review

```json
{
  "label": "Workspace Copilot",
  "agent": "workspace-copilot",
  "sessionPrefix": "workspace-copilot",
  "outputRoot": "./reports",
  "cron": {
    "enabled": true,
    "schedule": "0 22 * * *",
    "launchPrompt": "./central scheduler prompt"
  }
}
```

Prompt for `./central scheduler prompt`:

```md
Review the workspace changes since the previous run. Summarize the highest-risk changes, open questions, missing tests, and next actions. Write the result to ./reports/nightly-review.md.
```

## Validate with `run-once`

```bash
kcron run-once
```

Use `run-once` before you trust a real schedule. It is the cleanest way to verify prompt behavior, output paths, and memory continuity without waiting for the next cron tick.

## Common automation patterns

| Pattern | Prompt focus | Durable output |
| --- | --- | --- |
| nightly repo review | risk, missing tests, next actions | `./reports/nightly-review.md` |
| deployment summary | deploy state, incidents, follow-ups | `./reports/operate-summary.md` |
| research digest | new findings, sources, open questions | `./reports/research-digest.md` |
| maintenance loop | outdated guidance, lint issues, follow-up work | `./reports/maintenance-pass.md` |

## Review the scheduled workflow

- Workspace Copilot now has a recurring nightly task
- the workspace produces predictable report artifacts in `./reports`
- interactive and scheduled runs now use the same workspace settings
- you can investigate a failure by inspecting the prompt, memory, report, and trace together

## Troubleshoot a scheduled run

- the workspace is not registered, so `kcron` never picks it up
- `cron.enabled` is `false`
- the launch prompt path is wrong
- the prompt writes outside `./reports`
- the task depends on an old session instead of workspace files and saved memory

## Make the automation understandable

The schedule is only one part of useful automation. You should also be able to review the prompt before it runs, find the output afterward, and understand what happened when the task does not complete.
