Kestrel Desktop is the flagship app in the Kestrel Suite. If you want the clearest product view of what Kestrel is for, start here.

Use Desktop when the workflow is rooted in local workspace state and operator control, and you want the primary packaged Kestrel experience instead of assembling the suite through the browser and terminal separately.

## Workflow

Desktop is the right surface when the user needs to:

- use the main Kestrel product surface for local work
- stay close to local workspace state
- run agent workflows without living in the terminal
- keep workspace catalog state and session continuity visible
- inspect runtime health, recovery, and operator actions in one place
- make Desktop the default evaluation and demo path for the suite

## Why start here

Desktop is the fastest way to understand the product story behind Kestrel:

- local workspace identity is visible
- persistent sessions and runtime health stay close at hand
- operator-friendly control does not require a browser-only workflow
- the broader suite stays available without competing for first-run attention

## How the rest of the suite supports it

- Kestrel One is the canonical hosted web product
- the CLI is the terminal companion for power users and operator flows
- Ruhroh owns evaluation execution outside this repository
- the SDK and package layer exist for teams extending or embedding the suite

## Supporting architecture

Desktop is implemented over the shared Kestrel runtime and Local Core. It remains independent from Kestrel One source and hosted configuration.

## First-run setup

Packaged Desktop starts with a guided setup flow that is meant to get a new install to a usable workspace without dropping into recovery first.

- The first step is an explicit provider choice: `openrouter`, `openai`, `anthropic`, `ollama`, or `lmstudio`.
- Hosted providers require an API key stored in Desktop settings.
- Local providers use local OpenAI-compatible endpoints and do not require an API key by default; `ollama` and `lmstudio` are beta paths for v0.5.
- Setup resumes at the first unfinished step if the user stops midway.
- Recovery screens are reserved for non-onboarding failures such as missing resources, database errors, or runtime boot failures.

## v0.5 beta install status

Kestrel Desktop `0.5.0-beta.0` is a macOS-first unsigned beta artifact. The beta proof path covers OpenRouter as the hosted-provider setup flow and Ollama as the local-provider setup flow. Auto-update, signed/notarized distribution, and polished Windows/Linux installer support are outside this beta.

Managed packaged Postgres, local providers, code/dev-shell workflows, and `kcron` automation should be treated as beta surfaces until their clean-machine behavior is proven.
