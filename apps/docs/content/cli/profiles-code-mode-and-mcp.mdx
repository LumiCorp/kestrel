Profiles are how the CLI decides what a run is allowed to do. They are not cosmetic presets. They shape the active model, tool allowlist, MCP connections, and code execution policy for the current session.

## Global versus workspace-local profiles

Use global profiles when the behavior belongs to you as an operator across many folders. Use workspace-local profiles when the behavior belongs to the project itself.

| Profile location | File | Best for |
| --- | --- | --- |
| global | `~/.kestrel/profiles.json` | personal defaults, reusable operator profiles, machine-wide MCP config |
| workspace-local | `global profile selection` | project-specific tools, prompts, scheduling, and local automation contracts |

If a workspace contains `global profile selection`, that workspace-local profile wins over a global profile while you are inside that folder.

## The fields that change runtime behavior

These are the capability-heavy profile fields worth understanding:

| Field | Purpose | Why it matters |
| --- | --- | --- |
| `toolAllowlist` | controls which tools can be used | narrows capability to the minimum needed for the task |
| `codeMode` | enables controlled code execution | switches the session from pure chat into code-capable behavior |
| `mcpServers` | configures external MCP-backed tools | lets the profile expose remote or local server tools |
| `sessionPrefix` | controls durable naming | makes session continuity readable across repeated runs |

## Example global profile

```json
{
  "version": 3,
  "profiles": [
    {
      "id": "reference",
      "label": "Reference React",
      "agent": "reference-react",
      "sessionPrefix": "reference",
      "toolAllowlist": ["free.hn.top", "free.time.current", "code.execute"],
      "codeMode": {
        "enabled": true,
        "languages": ["javascript", "python", "bash"],
        "sandbox": {
          "executor": "docker",
          "timeoutMs": 20000,
          "memoryMb": 256,
          "cpuShares": 256,
          "networkDefault": "off",
          "allowDependencyInstall": false
        },
        "retention": {
          "persistSummary": true,
          "persistArtifacts": true
        },
        "approvalMode": "auto"
      },
      "mcpServers": [],
      "default": true
    }
  ]
}
```

## Workspace-local example

The Workspace Copilot flow is stronger when the profile lives with the project:

```json
{
  "id": "workspace-copilot",
  "label": "Workspace Copilot",
  "agent": "workspace-copilot",
  "sessionPrefix": "workspace-copilot",
  "toolAllowlist": ["fs.read_text", "code.execute"],
  "mcpServers": [
    {
      "id": "docker-gateway",
      "transport": "stdio",
      "command": "docker",
      "args": ["run", "--rm", "mcp/gateway"]
    }
  ],
  "codeMode": {
    "enabled": true,
    "languages": ["bash", "typescript"],
    "sandbox": {
      "executor": "docker",
      "timeoutMs": 20000,
      "memoryMb": 256,
      "cpuShares": 256,
      "networkDefault": "off",
      "allowDependencyInstall": false
    },
    "retention": {
      "persistSummary": true,
      "persistArtifacts": true
    },
    "approvalMode": "auto"
  }
}
```

## Tool allowlists

`toolAllowlist` limits which tools the profile may use. Include only what the workflow needs.

- add only the tools the workflow needs
- expect `code.execute` to be present when `codeMode.enabled` is `true`
- remove MCP-provided tools when you remove the server that owns them

The CLI merges and normalizes tool allowlists during profile loading. That means a profile is the durable policy document, while the runtime sees an explicit effective allowlist.

## Code mode

Code mode is beta for Kestrel v0.5. It is useful when a workflow genuinely needs local execution, but it is not part of the default Desktop first-run promise.

`/code` is not a theme toggle. It changes runtime behavior:

```text
/code status
/code policy
/code enable
/code disable
```

Use code mode when the workflow needs to execute code. Keep it off for review-only or writing tasks. When enabled, Kestrel validates the sandbox settings and adds `code.execute` to the allowed tools.

## MCP servers

`mcpServers` lets a profile expose remote or local capability bundles through MCP. The CLI supports:

```text
/mcp status
/mcp servers
/mcp tools [serverId]
/mcp add stdio <id> <command> [args...]
/mcp add http <id> <url> [--auth-env VAR] [--header-env Name=ENV]
/mcp add sse <id> <url> [--auth-env VAR] [--header-env Name=ENV]
/mcp docker [server]
/mcp allow <toolId>
/mcp deny <toolId>
/mcp remove <serverId>
/mcp refresh
```

Use MCP when a profile or workspace needs a named external tool connection.

## Review the resulting profile

This is the point where Kestrel stops being “a terminal chat that happens to run tools” and becomes a configurable local runtime. Profiles let the same binary behave differently for narrative chat, code tasks, workspace automation, and MCP-backed operator flows.

## Keep capabilities explicit

Profiles keep model, tool, MCP, and sandbox choices visible. A workspace-local profile stores those choices with the project instead of relying on settings from one person's home directory.
