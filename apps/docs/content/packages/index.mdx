The packages section is where Kestrel becomes something you can embed. These packages are deliberately narrow, but they now cover multiple integration depths: compatibility HTTP, typed agent APIs, lower-level control-plane access, and observability.

This section is for product engineers choosing the right integration surface.

## Start here if you are...

- keeping an OpenAI-style app contract while moving to Kestrel: read [OpenAI-compatible HTTP](/build/openai-compatible-http)
- creating a typed server-side agent integration: read [SDK](/packages/sdk)
- wiring application routes in Next.js: read [Next.js](/packages/next)
- instrumenting runs, streams, and subscriptions: read [Observability](/packages/observability)

## How to choose

| Need | Start here |
| --- | --- |
| keep existing chat/responses clients | [OpenAI-compatible HTTP](/build/openai-compatible-http) |
| build server-owned product logic | [SDK](/packages/sdk) |
| expose typed Next.js routes | [Next.js](/packages/next) |
| inspect task graph, project snapshot, or review state | [SDK](/packages/sdk) then [Review and state workflows](/operate/review-and-state) |
| add traces and OTEL export | [Observability](/packages/observability) |
