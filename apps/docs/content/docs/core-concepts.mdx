Kestrel becomes much easier to read once you stop treating it as "the chat app" or "the SDK." The same concepts appear in the runtime, packages, CLI, Kestrel One, Desktop, and workspace automation flows. This page names those concepts so the rest of the docs can stay specific without repeating the same definitions.

## Runner service

The runner service is the HTTP boundary that executes Kestrel work on behalf of applications. Product apps and backend services call it. Browsers do not. That boundary is what lets the SDK, Next.js helpers, OpenAI-compatible HTTP routes, and operator applications share one execution model without copying runtime internals into every surface.

## Agent

An agent is the application-facing definition created with `createAgent()`. It binds an `id`, a `profileId`, runner-service credentials, and methods such as `run`, `stream`, `resume`, `session`, and `subscribe`. It is the cleanest package-level expression of the runtime contract.

## Session

A session is the durable continuity boundary for a conversation, task stream, or recurring workflow. When you pass `sessionId` into the SDK or the Next.js route helpers, you are telling Kestrel which history, memory, and replay line should receive the next turn.

## Memory

Kestrel exposes explicit session memory instead of hidden state. The SDK reads memory with `session().memory.get()` and updates it with `session().memory.update({ expectedRevision, patch })`. That versioned update model gives applications a stable way to record task context without bypassing the runtime's evidence trail or silently overwriting parallel work.

## Stream versus subscription

A request stream is scoped to one run request. A subscription is a separate background event stream scoped by `sessionId`, `threadId`, or `runId`. Replay is different again: it is the durable historical record after the run is over. The root SDK does not expose a global unfiltered event firehose because those three jobs should not collapse into one mechanism.

## Replay

Replay means the system keeps enough durable evidence to inspect what happened after the run completes. That shows up in run events, append-only CLI history, memory revisions, and the operator workflows around migrations, runbooks, and scorecards.

## Operator control

Operator control is the part of the system that lets a human work with a run that already exists. `operator.inbox`, `operator.thread`, and `operator.control` are the primitives behind the CLI, the web workbench, and the SDK runner client. They matter because Kestrel is meant to support steering and inspection after a run begins, not only request initiation.

## Workspace

A workspace is a machine-local catalog entry for a project root. Kestrel resolves the nearest Git root when it can, otherwise it uses the current directory, and records that identity in `~/.kestrel/workspaces.json`.

## Automation

Workspace discovery does not imply recurring execution. The catalog has an `automationEnabled` flag for the future central scheduler, while runtime behavior continues to come from the selected global profile and active session.

## Quality gates

The repo treats governance checks, tests, prompt suite runs, and released Ruhroh validation as part of the product, not just internal hygiene. The runtime model is only useful if changes remain inspectable and reproducible.
