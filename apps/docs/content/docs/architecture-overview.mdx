Kestrel separates the product a person uses from the runtime that executes agent work. That separation lets Desktop, Kestrel One, the SDK, Next.js helpers, and compatible HTTP routes share the same run and result behavior.

## The request path

```text
person or browser
  → Desktop or your application server
  → authenticated runner service
  → durable runtime
  → model and allowed tools
  → events, terminal result, and persisted evidence
```

Browser code never needs the runner token or provider credentials. The product server establishes the user, organization, session, model access, and other trusted context before asking the runner to execute.

## The durable runtime

The runtime coordinates run and step lifecycle, model and tool input/output, persistence, waiting, cancellation, events, artifacts, and replay evidence. It does not need to know which user interface initiated the request.

## Product and integration surfaces

- Desktop packages a local application, Local Core, and managed persistence.
- Kestrel One owns hosted accounts, organizations, Projects, Threads, Knowledge, and model access.
- The SDK provides typed server-side agent and session methods.
- Next.js helpers adapt those methods to application routes.
- OpenAI-compatible HTTP supports applications that already use familiar request shapes.
- The CLI exposes local and operational controls.
